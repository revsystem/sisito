## 捨てた選択肢と理由

- Rails 8.x への即時メジャー更新: Propshaft デフォルト化・Solid Cache/Queue/Cable 導入など影響範囲が広く短期修正の範囲外 — 7.2.3.1 パッチ更新を採用
- `dartsass-rails`: Sprockets 非対応で bootstrap-sass と衝突、assets:precompile が失敗する既知問題 — `dartsass-sprockets`（sassc-rails API 互換フォーク）を採用
- `propshaft` 移行: アセットパイプライン全面刷新、Sprockets 明示設定済みのアプリに対してリスク高 — Sprockets 維持
- turbolinks → turbo-rails 移行: JS 連携書き換えが必要、本計画の short-term 範囲を超える — data 属性削除のみで完全除去を選択
- CLAUDE.md への追記（utility scripts の記載）: 承認前の段階で次タスクに移行したため保留状態

## ハマりどころ

- `/usr/bin/ruby3.2: no such file or directory` で bundle install 失敗 → プロジェクトに Ruby 環境が未整備 → mise でプロジェクトスコープ管理に移行して解消
- mysql2 gem ネイティブ拡張ビルド失敗 `checking for -lmysqlclient... no` → libmysqlclient-dev / libmariadb-dev が未インストール → ユーザーが sudo で libmysqlclient-dev をインストールして解消
- bundle-audit が `bundle audit` ではなく `bundle-audit` コマンドで起動する点（gem install bundler-audit で提供）

## 学び

- bundler-audit 0.9.3 の ruby-advisory-db は 2026-03-30 時点で 1078 advisories を収録。`--update` で常に最新化
- Rails 7.2.x は 2026-08-09 で EOL（security only）、残り約 4 ヶ月
- Rails 8.1.3 が 2026-03-24 リリース済み、8.1 は Ruby 3.2.0+ を要求
- `rails '~> 7.2', '>= 7.2.2'` は 7.2.x 系パッチを自動取り込み可能（制約緩和不要、`bundle update rails` のみで 7.2.3.1 に上がる）
- mise は `mise use ruby@X.Y.Z` でカレントディレクトリに mise.toml を生成しプロジェクトスコープで Ruby 固定できる
- bootstrap-sass 3.4 は LibSass 依存で `@extend .col-sm-6` 等の Bootstrap クラス extend を多用、置換時は Dart Sass 互換性確認必須
