## 決定事項

- Rails のメジャー更新（8.x）は本計画に含めない。Rails 7.2.x は 2026-08-09 で EOL だが短期修正スコープを優先
- Sass の置換先は `dartsass-rails` ではなく `dartsass-sprockets` を採用。Sprockets 連携と bootstrap-sass 3.4 互換を維持するため

- CI には `bundler-audit` を追加。Dependabot は別途検討

## 捨てた選択肢と理由

- Rails 8.x への即時メジャー更新: Propshaft デフォルト化 / Solid Cache・Queue・Cable 導入など影響範囲が広く短期修正の範囲を超える — 7.2.3.1 パッチ更新に留めた
- `dartsass-rails`: Sprockets 非対応で bootstrap-sass と衝突、assets:precompile が失敗する既知問題 — `dartsass-sprockets`（sassc-rails API 互換フォーク）を採用
- `propshaft` への移行: アセットパイプライン全面刷新。Sprockets 明示設定済みのアプリに対してリスクが大きい — Sprockets 維持
- turbolinks → turbo-rails 移行: JS 連携書き換えが必要で short-term の範囲外 — layout の data 属性削除のみで完全除去
- `feature/security-patch-and-deprecated-gem-removal` ブランチ名: ユーザー指摘で既存規則 `heads/Rails_vX.Y.Z` に合わせて `heads/Rails_v7.2.3.1` に改名
- Issue を PR より先に作成する global CLAUDE.md の方針: 初回 `gh issue create` で `repository has disabled issues` エラー。ユーザー指示で Issues を有効化した上で Issue → PR 紐付けに戻した

## ハマりどころ

- `mise exec -- bundle exec rails test` が `Spring reloads, and therefore needs the application to have reloading enabled` で失敗 → Spring と `config.cache_classes = true` の競合 → `DISABLE_SPRING=1` で回避したが、その先で `Can't connect to local MySQL server` エラー → WSL 環境に MySQL サーバーがない → テストは Docker 環境に委ねる方針で `rails runner` によるブート確認に切り替えた
- `rm app/assets/javascripts/*.coffee` がインタラクティブ確認モードで停止 → Bash ツールは非対話実行のためプロンプトで無限待機 → `rm -f` に切り替えて解消
- `gh issue create` が `repository has disabled issues` で失敗 → リポジトリ設定で Issues が無効化されていた → `gh api -X PATCH repos/... -f has_issues=true` で有効化して解消
- `bundle install` 初回失敗 `/usr/bin/ruby3.2: no such file or directory` → Ruby 未インストール → mise でプロジェクト固定した上でさらに mysql2 ビルド失敗 → `sudo apt-get install libmysqlclient-dev` で解消（ユーザー側で手動実行）

## 学び

- Rails 7.2 + Spring + `config.cache_classes=true` はテストで衝突する。CI で Spring を使う場合は `DISABLE_SPRING=1` か test.rb 側を `cache_classes=false` に調整が必要
- `gh pr create` は Issue 未作成でも通る。後から `gh pr edit --body-file` で `Closes #N` を先頭に追記すれば紐付け可能
- `gh api -X PATCH repos/OWNER/REPO -f has_issues=true` でリポジトリ設定の Issues 有効化が可能（Settings 画面操作不要）
- `mise use node@22` 実行時に LTS 最新 patch（今回は 22.22.2）が自動選択される。設定ファイルには `node = "22"` のみ記録される
- `dartsass-sprockets` は `sassc-rails` の API 互換フォーク。Gemfile の差し替えのみで既存 SCSS（`@extend`、`@import`）はそのまま動作する
- `bundler-audit check --update` はオフラインでも動作可能（`--update` を省くと DB 更新を省略）
- Rails 7.2 系のパッチ制約は `'~> 7.2', '>= 7.2.2'` のままでも `bundle update rails` で 7.2.3.1 まで自動取り込み可能。Gemfile の下限更新は必須ではないが明示した
