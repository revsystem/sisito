## 決定事項

- `bin/deploy.sh` から `db:migrate` を完全に外す。マイグレ適用は人間がメンテナンスウィンドウで明示的に実行する。理由: `RAILS_ENV=production` だと `secret_key_base` 欠落で `db:migrate` が即落ちする上、deploy ごとに自動適用するのは migration の性質上危険
- Pi のデフォルト動作モードは `RAILS_ENV=development`。`config/database.yml` の `production` ブロックは存在しない `sisito_production` を参照しているため事実上未使用。`bin/deploy.sh` も `RAILS_ENV` を `development` デフォルトにし、override したい場合のみ環境変数で指定する形にした
- bundler の deprecated 警告対策として、`--deployment` / `--without` フラグではなく `bundle config set --local <key> <value>` を使う
- `bundle config set --local without 'test'` のみにして `development` グループは含めない。`config/environments/development.rb` が `listen` gem に依存しており、`without 'development test'` だと `assets:precompile` が `Gem::LoadError: listen is not part of the bundle` で落ちる
- 性能インデックスの整理は **既存 `20250705*` migration を書き換える方式ではなく、新規 cleanup migration を別ファイルで追加する方式** を採用（→「捨てた選択肢」参照）
- PR #15 では `db/schema.rb` 更新と `CLAUDE.md` Gotcha #2 削除を **同一 PR にまとめる**。当初は別 PR に分ける案だったが、schema.rb を更新したのに「Schema version lag」Gotcha が残るのは状態と矛盾するため
- セッションをまたいで永続化したい todo は、独立ファイルではなく **このハンドオーバーノート（または次セッションの handover）に直接書く**。理由: 独立 todo ファイルを作っても結局 handover を読むタイミングと同期しないため、handover に集約した方が見落としにくい
- **【次セッション以降の標準ワークフロー】Pi で動作する変更（migration、deploy 関連、本番設定など）は、原則として「ローカルで修正 → Pi へ SCP で反映 → Pi 上で動作確認 → 期待通りなら 1 コミットにまとめて push & PR 作成」のループで進める。試行錯誤の commit や force-push を remote に積み重ねない。**
  - 今回 PR #15 では「ローカルで commit → push → Pi で検証 → 方針転換 → `git reset --mixed HEAD~1` → 再構築 → `force-with-lease` で push し直し」という手戻りが発生した。検証前にコミットしないことで、この種の reset / force-push を構造的に避けられる
  - PR タイトル / 本文を含めて確定してから 1 commit + 1 push で出すと、PR 履歴も clean なまま残る
  - 例外: ローカルで完結する純粋な refactor やドキュメント更新など、Pi の状態に依存しないものはこの限りではない

## 捨てた選択肢と理由

- **`bin/deploy.sh` で `db:migrate` を自動実行**: 初回 Pi 検証で `RAILS_ENV=production` だと `Missing 'secret_key_base'` で即死、deploy 自動化と migration 適用を分離すべきという判断 — `db:migrate:status` の出力をパースして保留があれば警告だけ出す形に変更（`SECRET_KEY_BASE_DUMMY=1` 経由）
- **`bundle config set --local without 'development test'`**: `assets:precompile` が `listen` gem の不在で落ちる — `without 'test'` のみに修正
- **既存 migration `20250705000001` を idempotent に書き換える + `20250705000002` を削除**: Pi の DB が「両 migration 適用済みだが手動で 3 本のインデックスを drop した」中途半端な状態になっており、また他のホスト（fresh DB）と Pi で migration の振る舞いが分岐するのを避けるべきという判断 — 既存 migration はオリジナルのまま温存し、新規 `20260425000001_cleanup_redundant_performance_indexes` を `if_exists` / `if_not_exists` で書いて全環境を同じ状態に収束させる方式に変更（→ 一度この方式で commit + push した PR #15 を `git reset --mixed HEAD~1` で取り消し、ファイルを master 状態に restore してから新方式で書き直し、`force-with-lease` で push し直した）
- **`db/schema.rb` 更新と Gotcha #2 削除を別 PR にする**: 状態とドキュメントが一時的に矛盾する — 同一 PR に統合
- **独立した todo 管理ファイル（`.claude/docs/follow-ups.md`）の運用**: 一度新設したが、そのファイルを読みに行くタイミングが結局 handover を読むタイミングと同じになり、handover と二重管理になる — 残 todo は handover ノート本文（次にやること）に統合する方式に一本化
- **MySQL の partial index (`add_index ..., where: ...`)**: そもそも MySQL 8 は `WHERE` 述語を黙って捨ててプレーンな BTREE を作るため partial index にならず、結果として既存インデックスと重複していた — `20260425000001` で 3 本とも drop し、CLAUDE.md の Gotcha に明記

## ハマりどころ

- `bin/deploy.sh` 初回実行で `db:migrate` が `Missing 'secret_key_base' for 'production' environment'` → `RAILS_ENV=production` がデフォルトになっていた + 鍵未設定 → `db:migrate` 自体を deploy.sh から外し、`db:migrate:status` を `SECRET_KEY_BASE_DUMMY=1` で叩いて警告だけ出す形に変更
- `bundle install` の deprecation 警告（`--deployment` / `--without` 廃止予定）→ `bundle config set --local <key> <value>` に置換
- `assets:precompile` が `Gem::LoadError: listen is not part of the bundle.` で失敗 → `bundle config set --local without 'development test'` で `:development` グループを除外していた → `config/environments/development.rb` の `EventedFileUpdateChecker` が `listen` を参照 → `without 'test'` のみに変更
- Pi 上で `bundle exec rails` が `command not found: rails` → ブランチ切り替え後 gem 実行ファイルが追従していなかった → `mise exec -- bundle install` で再整合
- 性能インデックス検証中、Pi の `monitor_performance.rb` で `idx_reason_destination` だけが `❌ 未作成` → controller のクエリを全部洗ったところ実は使っているのに DB から手動で消されていたことが判明 → `20260425000001` で `add_index ... if_not_exists: true` を入れて復活
- Pi での `git pull --ff-only` が `local changes would be overwritten by merge` で停止 → scp で送った 3 ファイル（merge 済み master と内容は完全一致だが git からは「未コミット変更」） → `git diff origin/master` で内容一致を確認した上で `git checkout --` + `rm` してから `git pull --ff-only` 成功
- ローカルでも一度 `19a377f` で「20250705000001 を書き換える」方式の commit をしていたところ、方針転換で `git reset --mixed HEAD~1` → `git checkout HEAD -- db/migrate/20250705000001_*.rb db/migrate/20250705000002_*.rb` で 2 ファイルだけ master 状態に戻し、それ以外（`monitor_performance.rb` / `CLAUDE.md`）は modified を維持、新規 migration を追加してから `force-with-lease` で push し直した

## 学び

- MySQL 8 は `add_index ..., where: ...` の `WHERE` 述語を黙って drop し、プレーンな BTREE インデックスを作る。Rails の partial index は MySQL では機能しない
- `add_index ... if_not_exists: true` と `remove_index ... if_exists: true` を組み合わせると、migration を「DB の現在状態に依らず最終状態に収束させる」冪等な形にできる。半端に手動操作が入った本番 DB を救済する用途で有効
- Pi の sisito は `RAILS_ENV=development` で動いている。`config/database.yml` の `production` ブロックが `sisito_production`（DB 不在）+ `root` 無パスワードを参照していて事実上未使用。`monitor_performance.rb` が `database: 'sisito_development'` をハードコードしているのもこのため
- `bundle config set --local without 'test'` で `:development` グループを残すのは Rails 開発時の `listen` gem 依存（`EventedFileUpdateChecker`）のため必要
- `db:schema:dump` 結果の `db/schema.rb` には schema 形式のバージョン（`ActiveRecord::Schema[7.2]`）と `collation: "utf8mb3_general_ci"` 等の charset/collation 明示が混入する。Rails 7.2 化のついでに最初に dump し直したタイミングで一気に反映される
- Cursor の `TodoWrite` で作る todo は Cursor チャットセッション内のメモリ管理で、Claude Code（および次の Cursor セッション）からは見えない。エージェント間・セッション間で引き継ぐには handover ノート等のファイルに書き出すしかない
- 「ローカル編集 → SCP → Pi で検証 → ループ」運用は、リモートの commit history を試行錯誤で汚さずに済む（PR #15 の force-push 1 回だけで決着）
- `git reset --mixed HEAD~1` 後、特定ファイルだけを HEAD 状態に戻すには `git checkout HEAD -- <files>` が手早い。残りは untracked / modified のまま残せる
